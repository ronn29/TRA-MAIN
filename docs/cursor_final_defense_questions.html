Answer 2: Class Weighting and Bias

We addressed bias through multiple mechanisms po:

Sa code (lines 138-146 in train.py), we're using a normalized weighting system: - Career weights are on a 0-3 scale (0=not relevant, 3=highly relevant) - We convert this to a multiplier (0.5 to 1.5) so no career gets completely excluded - Weights are then normalized to the maximum weight

Even if a career has low relevance scores, it still gets a weight of at least 0.5 relative to the max weight. The model uses class_weight='balanced' as fallback kung walang weights available, which automatically adjusts for class imbalance. This ensures rare but valid careers can still be recommended kung yung student's answers strongly match that career pattern.

Answer 3: Feature Engineering and Question Weights

The question weights were designed based on question relevance sa different programs po.

Looking at lines 194-217 in train.py: - We have two types of weights: universal (same for all programs) and program-specific (dict with different weights per program code) - Program-specific questions (like CS_Q1-Q10, IT_Q1-Q10, etc.) naturally have higher weights when matched to their program - General questions (Apt_Q, Int_Q, Pers_Q) have universal weights

The vectorized approach ensures efficient processing - instead of row-by-row, we map weights using pandas Series operations which is much faster for large datasets.

Unfortunately, we haven't documented a formal feature importance validation study yet, pero the RandomForest model itself has feature_importances_ attribute (lines 277-285) that we print after training to see which features actually matter most.

Current approach is defensible IF: - We're treating -1 as "not applicable, ignore" signal - Model learns to ignore these columns via low feature importance - We verify that N/A columns don't leak information
